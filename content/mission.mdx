<h2 id="h-problem">The problem we set out to solve</h2>

<div className="callout note"><div className="ic">ⓘ</div><p>Nomba exposes payment primitives but does not ship a managed subscriptions layer, so teams repeatedly rebuild this capability from scratch.</p></div>

That's the brief's own framing of the gap, and it's a narrower, sharper claim than it first sounds. Nomba already has the primitives a subscriptions product needs, Checkout, tokenised cards, a Charge API, Transfers. What's missing isn't payment capability. It's everything <em>around</em> the charge: the state machine that knows a subscription is <code>past_due</code> and not just <code>failed</code>, the recovery workflow that knows to retry before it suspends, the invoice that knows how to prorate a mid-cycle upgrade, the webhook that a downstream team can trust enough to build on.

Every team that wants recurring billing on Nomba today either builds that layer themselves, badly, on a deadline, once per team, or reaches for a subscriptions tool that wasn't built for how Nigerian subscribers actually pay and communicate: cards decline more often than global averages, and a meaningful share of customers are more reachable by SMS or USSD than by an app push notification or an email that lands in a folder nobody opens.

This build is that missing layer, built once, so other teams don't have to build it again.

<h2 id="h-criteria">How we hit each judging criterion</h2>

<p>The track is judged on four things. Here's where each one surfaces in the system, not just in this document.</p>

<p><strong>State-machine completeness</strong><br />
The subscription lifecycle is an explicit, enforced state machine, <code>pending → trialing → active → past_due → grace_period → suspended → cancelled / expired</code>, not a status string set ad hoc from scattered call sites. Every transition is a domain event (<code>SubscriptionCreated</code>, <code>SubscriptionUpdated</code>, <code>SubscriptionCancelled</code>, <code>SubscriptionRenewed</code>, <code>PaymentFailed</code>, <code>PaymentRecovered</code>, <code>InvoicePaid</code>) persisted to an append-only event store, which is also what the webhook system and analytics read from, one system of record, not three.</p>

<p><strong>Dunning sophistication</strong><br />
A failed charge doesn't just retry on a timer. It opens a grace period and reaches the customer on whichever channel they're actually on: a WhatsApp message with inline Retry / Pause / Downgrade actions, processed straight from the reply; SMS as the guaranteed fallback if WhatsApp isn't available for that customer; USSD for feature-phone and low-data customers who can check status, pause, or cancel from a menu with no data connection at all. Every channel a customer's action arrives on is logged, which is what makes channel-level recovery-rate reporting possible instead of a single blended number.</p>

<p><strong>Multi-tenant cleanliness</strong><br />
Every business entity, plans, customers, subscriptions, invoices, payments, webhooks, API keys, audit logs, carries a <code>merchantId</code> inherited from a shared base entity, enforced at the data-access layer rather than trusted to application-level filtering. Two merchants running simultaneously never see a row that isn't theirs, by construction, not by convention.</p>

<p><strong>API ergonomics for downstream developers</strong><br />
Every authenticated endpoint takes a Bearer token; JWT payloads carry the <code>merchantId</code> that scopes everything downstream of login. Responses follow one consistent envelope. Webhook payloads are signed so an integrator can verify authenticity without a support call, replayed deliveries are idempotent so reprocessing never duplicates a side effect, and the full API surface is documented and testable directly through Swagger, a developer unfamiliar with the system should be able to complete a standard integration from the docs alone.</p>
